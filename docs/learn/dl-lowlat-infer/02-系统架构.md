## 分层架构说明

### 第1层：应用入口

**`main.cu`** 是程序入口，负责：
- 解析命令行参数（模型名称、算法、精度、运行时长等）
- 根据参数创建 `Benchmark` 对象
- 启动基准测试

### 第2层：基准测试框架

**`Benchmark`** 是测试的编排者：
- 创建 1 到 N 个 `ModelInstance`（支持多实例并发）
- 每个实例运行在独立的线程和 CUDA 绿色上下文中
- 收集所有实例的延迟结果，计算统计数据（平均值、P99 等）

### 第3层：模型实例

**`ModelInstance`** 是单个推理任务的管理者：
```
循环执行：
  1. 从 DataIterator 取下一个输入
  2. 记录开始时间
  3. 调用 Model.Input() 把输入复制到 GPU
  4. 调用 Model.LstmForward() 触发推理
  5. 调用 Model.OutputWait() 等待结果
  6. 调用 Model.Output() 取回结果
  7. 记录结束时间
  8. 把结果存入 InferenceResult
```

### 第4层：模型实现（关键）

这一层有**两个技术路线**，代表了两种截然不同的延迟优化策略：

#### 路线 A：ModelSingleStepBase（cuBLAS 实现）

```
CPU                              GPU
 │                                │
 │ ── 复制输入数据 ──────────────► │
 │                                │
 │ ── 启动 cuBLAS GEMV kernel ──► │
 │                                │ ← 执行矩阵-向量乘法
 │                                │ ← 激活函数
 │                                │ ← 更新状态
 │ ◄── 等待 CUDA 同步 ──────────── │
 │                                │
 │ ── 聚合（点积+偏置）──────────► │
 │ ◄── 等待结果 ───────────────── │
```

**特点**：每次推理都要启动新的 kernel，有约 5-10µs 的 kernel 启动开销。

#### 路线 B：ModelPersistentBase（持久化内核实现）

```
【初始化阶段，只做一次】
CPU                              GPU
 │                                │
 │ ── 启动持久化 kernel ─────────► │
 │                                │ ← 加载权重到共享内存
 │                                │ ← 发出"我已就绪"信号
 │ ◄── 收到就绪信号 ──────────── │
 │                                │ ← kernel 进入等待循环
 │                                │   （自旋等待下一个输入）

【每次推理（微秒级）】
CPU                              GPU
 │                                │
 │ ── 复制输入数据 ──────────────► │
 │ ── 原子写入"开始"信号 ─────────► │
 │                                │ ← 检测到信号，读取输入
 │                                │ ← 执行 LSTM 计算
 │                                │ ← 原子写入输出值（兼作完成信号）
 │ ◄── 原子读到输出值 ──────────── │
```

**特点**：kernel 始终在运行，消除了 kernel 启动开销，只需原子操作同步。

## 数据流架构

```
磁盘文件 (.npy)
    │
    ▼
ModelDef（模型权重定义）
    │ 加载权重矩阵 W, U, B
    ▼
GPU 显存
    │
    ├── 权重矩阵（一次性加载，永久存在）
    │       W[l], U[l], B[l]  （每层一组）
    │
    ├── 状态缓冲区（每步更新）
    │       h_t[l][s]   隐藏状态滑动窗口
    │       c_t[l][s]   细胞状态滑动窗口
    │
    └── 输入/输出缓冲区（每步覆写）
            dev_x（输入向量）
            dev_y（输出向量或标量）
```

## 多实例架构（CUDA 绿色上下文）

```
┌─────────────────────────────────────────────────────┐
│                    单块 GPU                          │
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐  │
│  │ GreenContext1 │  │ GreenContext2 │  │   ...    │  │
│  │              │  │              │  │          │  │
│  │  2个SM分区   │  │   2个SM分区  │  │ 2个SM分区 │  │
│  │              │  │              │  │          │  │
│  │  ModelInst.1 │  │  ModelInst.2 │  │  Inst.N  │  │
│  └──────────────┘  └──────────────┘  └──────────┘  │
│                                                     │
│  CPU线程1           CPU线程2          CPU线程N       │
└─────────────────────────────────────────────────────┘
```

**绿色上下文**是 CUDA 12.0+ 的新特性，允许一块 GPU 被划分成多个"虚拟分区"，每个分区有固定的 SM（流多处理器）资源配额，相互隔离不干扰。

- `lstm_s`：每实例 2 个 SM，可以同时运行 8 个实例（假设 GPU 有 16 个 SM）
- `lstm_m`：每实例最多 16 个 SM，可以同时运行 4 个实例
- `lstm_l`：占用整块 GPU，只能运行 1 个实例

## 关键设计决策

### 1. 为什么使用持久化内核？

普通 CUDA kernel 每次启动需要约 5-10µs 的调度开销。对于 lstm_s（目标延迟 < 10µs），这个开销是不可接受的。持久化内核消除了这个开销。

### 2. 为什么用原子操作同步而不是 CUDA 事件？

CUDA 事件（`cudaEventRecord`/`cudaEventSynchronize`）也有微秒级的开销，且走的是 CUDA 驱动路径。原子操作直接在共享内存/pinned 内存上操作，是最低开销的同步方式。

### 3. 为什么用 GDRCopy？

在 PCIe 系统上，CPU 轮询 GPU 内存（pinned memory）需要经过 PCIe 总线，有约 1µs 的延迟。GDRCopy 通过 GPUDirect RDMA 在 CPU 地址空间映射 GPU 内存，使 CPU 可以零拷贝、低延迟地读写 GPU 内存，节省约 0.5µs。
