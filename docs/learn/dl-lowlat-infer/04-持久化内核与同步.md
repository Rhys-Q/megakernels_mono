# 持久化内核与同步机制

> 这是本项目最核心、最精妙的技术。理解了这部分，就理解了为什么它能达到微秒级延迟。

## 核心思想

### 普通 kernel 的问题

在普通 CUDA 编程中，每次想让 GPU 做一点计算，需要：

```
CPU                              GPU
 │                                │
 │  1. 准备参数（几微秒）           │
 │  2. 调用 cudaLaunchKernel()     │
 │     ↓ 经过驱动、调度器           │
 │                                │  3. SM 调度，线程初始化
 │                                │  4. 从显存加载权重到寄存器/cache
 │                                │  5. 执行计算
 │                                │  6. 退出 kernel
 │  7. 等待同步，读结果             │
```

其中步骤 1-3（kernel 启动）大概需要 **5-10 微秒**。对于 lstm_s 这种极小的计算（实际计算只需 ~1µs），光启动开销就是计算时间的数倍。

### 持久化 kernel 的解决方案

```
【启动阶段（只做一次）】
CPU: 启动 kernel → GPU: 加载权重 → GPU: 通知"我准备好了" → CPU: 收到

【每次推理（只有 ~1µs）】
CPU: 写入"开始"信号 → GPU: 检测到信号，计算 → GPU: 写入输出值 → CPU: 读到输出
```

kernel 永远不退出，权重永远在共享内存/寄存器里，每次推理只有信号传递的开销。

## CPU-GPU 同步：三个原语

所有同步都通过共享的**pinned 内存**（也叫 page-locked memory）进行。Pinned 内存是一种特殊的主机内存，GPU 可以直接读写它（无需经过 PCIe DMA 控制器）。

### 原语 1：GPU 通知 CPU"我已就绪"

**时机**：kernel 刚启动，权重加载完毕。

**函数**：`gpu_cpu_sync_part_0()`

```cuda
// GPU 端（只有 thread rank 0 执行）
template <typename group_type>
__device__ void gpu_cpu_sync_part_0(volatile int* signal, group_type& group) {
    group.sync();  // 等待所有线程加载完权重
    if (group.thread_rank() == 0) {
        // 向 CPU 可见的 pinned memory 写入"就绪"信号值
        store_atomic_relaxed(signal, HOST_SPIN_0_UNTIL_SIGNAL_VALUE);
    }
}
```

**CPU 端**（在 `PreProcess()` 中）：
```cpp
// CPU 自旋等待，直到 GPU 写入就绪信号
while (host_signal != HOST_SPIN_0_UNTIL_SIGNAL_VALUE) {
    // 忙等待（busy-wait），延迟极低
}
```

---

### 原语 2：GPU 等待 CPU"开始"信号

**时机**：每次推理开始前，GPU 等待 CPU 提供新输入并发出开始信号。

这里分**两种架构**，因为它们的硬件特性不同：

#### Grace Hopper 系统（AArch64，统一内存）

```cuda
// 函数：gpu_cpu_sync_part_1()
// 使用 compare-exchange（比较并交换）原子操作

__device__ void gpu_cpu_sync_part_1(volatile int* signal, group_type& group) {
    if (group.thread_rank() == 0) {
        // 循环尝试：期望信号值为 DEVICE_SPIN_0_UNTIL_SIGNAL_VALUE
        // 如果匹配，原子地将其改为 0（同时读走信号，防止重复触发）
        poll_atomic(signal, 
                    DEVICE_SPIN_0_UNTIL_SIGNAL_VALUE,  // 期望值
                    0u);                               // 重置为 0
    }
    group.sync();  // 等待 thread rank 0 的结果广播到所有线程
}
```

这种方式之所以能在 Grace Hopper 上用，是因为 CPU 和 GPU 共享物理内存（NVLink-C2C），`compare_exchange_strong` 可以跨 CPU/GPU 原子工作。

#### PCIe 系统（x86_64）

```cuda
// 函数：gpu_cpu_sync_part_1_no_aex()
// 使用"开关值交替"机制，避免 compare-exchange 的问题

__device__ void gpu_cpu_sync_part_1_no_aex(volatile int* signal, 
                                             group_type& group,
                                             volatile uint32_t* device_spin_switch) {
    if (group.thread_rank() == 0) {
        // 等待信号值等于 SIGNAL_VALUE * device_spin_switch
        // device_spin_switch 在 0 和 1 之间交替
        poll_load(signal, 
                  DEVICE_SPIN_0_UNTIL_SIGNAL_VALUE * *device_spin_switch);
        
        // 收到信号后，翻转开关，为下次准备
        *device_spin_switch = !(*device_spin_switch);
    }
    group.sync();
}
```

为什么要用"开关"交替？因为 PCIe 系统上无法原子地"读取并清零"，CPU 写入信号后，GPU 读到了，但信号还在内存里。下次推理时，GPU 会立刻再次读到相同的值，造成错误。

通过让信号值在奇偶次推理之间交替（比如 1、0、1、0...），GPU 只接受符合当前"开关值"的信号，避免混淆。

---

### 原语 3：GPU 通知 CPU"计算完成"（兼输出值）

**时机**：每次 LSTM 计算结束。

**函数**：`gpu_cpu_sync_part_2()`

```cuda
// GPU 端（只有 thread rank 0 执行）
template <typename group_type>
__device__ void gpu_cpu_sync_part_2(int* signal, float value, group_type& group) {
    if (group.thread_rank() == 0) {
        // 把浮点输出值写入 CPU 可见的 pinned memory
        // 这个写入本身就是完成信号！
        store_atomic_relaxed((float*)signal, value);
    }
}
```

**CPU 端**（在 `Output()` 中）：
```cpp
// CPU 自旋等待，直到出现非 NaN 的有效输出值
float output;
do {
    output = *(volatile float*)host_signal;
} while (isnan(output));

// 读到有效值，就是推理结果
return output;
```

**妙处**：输出值和完成信号是**同一个值**！GPU 把计算结果直接写入信号位置，CPU 检测到非 NaN 值就知道计算完成了，同时也拿到了结果。

## 完整同步时序图

```
时间轴 ──────────────────────────────────────────────────────────►

【初始化（一次性）】

CPU: ────────────────── 等待就绪信号 ──────────────────────►
GPU: ──► 加载权重 ──► 写"就绪" ─────────────────────────────►
                      ↑
                    信号传递

【第 1 次推理】

CPU: ──► 复制输入 ──► 写"开始"信号 ──────────────── 读到输出值 ──►
GPU: ───────────────────────────────► 检测开始 ──► 计算 ──► 写输出 ──►
                                        ↑                   ↑
                                      信号1                信号2

【第 2 次推理】

CPU: ──► 复制输入 ──► 写"开始"信号 ──────────────── 读到输出值 ──►
GPU: ───────────────────────────────► 检测开始 ──► 计算 ──► 写输出 ──►
```

## 三种 kernel 并行规模

根据模型大小，使用不同规模的并行来做矩阵-向量乘法：

### lstm_s：单 Block（`gemv_lstm_bp_impl.cu`）

```
一个 CUDA Block（含若干 warp）
  ┌─────────────────────────────────┐
  │  Warp 0: 负责输出向量的第 0-31 个元素  │
  │  Warp 1: 负责输出向量的第 32-63 个元素 │
  │  ...                             │
  │  Warp N: 负责输出向量的第 ... 个元素   │
  └─────────────────────────────────┘
  所有 warp 协作完成一次 GEMV
```

单 block 可以直接用 `__syncthreads()` 同步，通信开销最小。

### lstm_m：线程块集群 Thread-Block-Cluster（`gemv_lstm_tbc_impl.cu`）

```
一个 Cluster（5 或 10 个 Block，取决于 GPU 架构）
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │  Block 0 │ │  Block 1 │ │  Block 2 │  ...
  │          │ │          │ │          │
  └──────────┘ └──────────┘ └──────────┘
  Cluster 内通过 mbarrier（或 cluster.sync()）同步
  可以访问彼此的共享内存（cluster shared memory）
```

TBC 是 CUDA Hopper 引入的新特性，允许多个 block 形成一个协作组，共享数据更高效。

`TBC_USE_MBARRIER=ON` 编译选项可启用更高效的 mbarrier 同步原语。

### lstm_l：设备级（`gemv_lstm_dev_impl.cu`）

```
几乎整个 GPU 的 SM 都参与计算
  ┌──────────────────────────────────────┐
  │  所有活跃 Block 协作完成 GEMV          │
  │  使用 Grid-Level Cooperative Groups  │
  │  需要 cudaLaunchCooperativeKernel()  │
  └──────────────────────────────────────┘
```

设备级 kernel 可以在 GPU 内部做 grid-level 同步，不需要退出 kernel 就能让所有线程协调工作。

## PCIe 开关管理详解

对于 PCIe 系统，需要额外维护一个"开关"变量来防止信号混淆：

```
第1次推理：
  CPU 写入信号值 = SIGNAL_VALUE × 1（开关=1）
  GPU 等待值 = SIGNAL_VALUE × 1  ✓ 匹配
  GPU 翻转开关：device_spin_switch = 0
  CPU 翻转开关：cpu_spin_switch = 0

第2次推理：
  CPU 写入信号值 = SIGNAL_VALUE × 0 = 0（开关=0）
  GPU 等待值 = SIGNAL_VALUE × 0 = 0  ✓ 匹配
  GPU 翻转开关：device_spin_switch = 1
  CPU 翻转开关：cpu_spin_switch = 1

第3次推理：
  CPU 写入信号值 = SIGNAL_VALUE × 1（开关=1）
  ...以此类推
```

**初始化**：开关从 1 开始（`SIGNAL_SWITCH_INIT_VALUE = 1`）。

当内核重新启动时（lstm_l 的情况），开关必须重置回初始值，否则 CPU 和 GPU 的开关会不一致。

## GDRCopy：更快的信号传递

在普通 PCIe 系统上，CPU 轮询 GPU pinned memory 信号时：

```
CPU polling → PCIe 总线 → 内存控制器 → pinned memory
```

这条路径有约 1µs 的往返延迟。

GDRCopy 使用 **GPUDirect RDMA** 技术，在 CPU 地址空间中直接映射一块 GPU 显存：

```
CPU polling → CPU 本地内存映射 → 直接读 GPU 显存
（绕过了 PCIe 往返，因为映射是持久的）
```

在 PCIe 系统上可节省约 **0.5µs** 延迟。

要使用 GDRCopy，需要：
1. 在主机上编译并加载 `gdrdrv` 内核模块
2. 运行时添加 `--gdrcopy` 标志

```bash
# 安装 gdrdrv
cd gdrdrv
./build_gdrcopy_driver.sh
./insmod.sh

# 运行时启用
nvLstmInf --gdrcopy lstm_s ./data/lstm_s ./data/lstm_s.npy 30
```

## 架构自动检测

代码在编译时自动检测系统架构，选择正确的同步策略：

```cpp
// 在 util.h 和 kernel_params.h 中定义
#ifdef AARCH64_GH
    // Grace Hopper：使用 compare-exchange
    #define DISABLE_ATOMIC_EXCHANGE 0
#else
    // PCIe（x86_64）：使用开关交替
    #define DISABLE_ATOMIC_EXCHANGE 1
#endif
```

所有条件编译分支在实际的 `gpu_cpu_sync_part_*` 函数中处理，上层代码无需关心平台差异。
