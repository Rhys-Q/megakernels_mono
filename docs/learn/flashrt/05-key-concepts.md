# FlashRT 关键技术概念

## 概念1：FP8 量化

### 为什么需要量化？

AI 模型的权重通常用 16 位浮点数（BF16）存储。一个 22 亿参数的模型需要：
```
22亿 × 2字节 = 4.4 GB
```

如果用 8 位（FP8）存储：
```
22亿 × 1字节 = 2.2 GB
```

内存减半，而且 8 位计算比 16 位计算快得多（GPU 的 FP8 计算单元吞吐量约为 FP16 的 2 倍）。

### FP8 E4M3 格式

FP8 有两种常见格式，FlashRT 使用 E4M3（Energy 4 bits Mantissa 3 bits）：

```
FP8 E4M3：1位符号 + 4位指数 + 3位尾数
         能表示的最大值：448
         精度：约 3 位有效数字

FP16：    1位符号 + 5位指数 + 10位尾数
         能表示的最大值：65504
         精度：约 4-5 位有效数字

BF16：    1位符号 + 8位指数 + 7位尾数
         能表示的最大值：~3.4×10^38
         精度：约 3 位有效数字
```

### 量化误差问题

FP8 的动态范围（能表示的最大值与最小非零值之比）远小于 BF16。如果直接把 BF16 转成 FP8，大的值会溢出，精度会大幅下降。

**解决方案**：缩放（Scaling）

```
实际激活值：[0.001, 0.5, 1.2, 50.0, 150.0]
最大值 = 150.0，FP8 E4M3 最大值 = 448.0

缩放因子 s = 448.0 / 150.0 = 2.99

量化：
  FP8_val = round(BF16_val / s)
  0.001  → round(0.001/2.99)  = round(0.000334) ≈ 0
  50.0   → round(50.0/2.99)   = round(16.7)    ≈ 17  → 反量化: 17 × 2.99 = 50.8
  150.0  → round(150.0/2.99)  = round(50.2)    ≈ 50  → 反量化: 50 × 2.99 = 149.5

误差很小！（对于大多数 AI 任务来说可以接受）
```

### 静态量化 vs 动态量化

**动态量化**：每次推理时计算当前批次的最大值。
- 优点：精度最高
- 缺点：每次推理都需要一次扫描操作，增加延迟

**静态量化**（FlashRT 使用）：提前收集大量数据的统计信息，确定一个固定的缩放因子。
- 优点：推理时不需要扫描，速度最快
- 缺点：如果新数据的分布与校准数据差异大，精度会下降

这就是为什么 FlashRT 需要"校准"步骤——通过几帧真实数据，确定每层的激活值范围。

### FP8 校准的注意事项

**百分位数 vs 最大值**：

使用最大值作为缩放基准会有问题：
```
假设 99 帧的激活值最大约为 100，
有 1 帧因为特殊场景，激活值达到了 5000。

如果用最大值 5000 作为基准：
  缩放因子 = 5000/448 = 11.16
  正常帧的值 100 → 100/11.16 = 8.96 → 9 → 反量化: 9×11.16 = 100.4  （还行）
  但精度损失了，因为 FP8 的 448 个格子中，
  只有约 8.96 个格子被用到了，而不是全部 448 个。

如果用 99.9 百分位数约 100 作为基准：
  缩放因子 = 100/448 = 0.223
  正常帧：100 → 100/0.223 ≈ 448 → 反量化：100   （精度最高）
  异常帧：5000 → 5000/0.223 ≈ 22421 → 截断到 448 → 反量化: 100   （溢出，但这帧本来就是异常）
```

FlashRT 默认使用 `percentile=99.9`，这样既保证了正常帧的最高精度，又不会被少数异常帧破坏量化质量。

## 概念2：CUDA Graph

### CUDA 的正常工作方式

没有 CUDA Graph 时，CPU 和 GPU 的交互大致是这样的：

```
CPU 时间线：
[Python代码] → [调用 cuBLAS GEMM] → [调用 RMSNorm内核] → [调用 cuBLAS GEMM] → ...
                 ↓                     ↓                     ↓
GPU 时间线：
             [等待]  [GEMM]  [空闲]  [RMSNorm]  [空闲]  [GEMM]  [空闲]  ...
```

问题：
1. GPU 在等 CPU 指令时是空闲的
2. 每次内核调用都有 ~5-10 微秒的 CPU 开销
3. 如果有 1000 个内核调用，就有 5-10 毫秒纯粹浪费在 CPU 调度上

### CUDA Graph 的工作方式

**录制阶段**（只做一次）：
```
GPU 进入录制模式 → 记录所有内核调用 → 构建执行图
                                        ↓
                                  [GEMM→RMSNorm→GEMM→...]
```

**重放阶段**（每次推理）：
```
CPU 发送一条指令：graph.replay()
GPU 收到指令后，自动按照图中的顺序执行所有操作
CPU 可以干其他事情（或者等待）
```

```
CPU 时间线：
[一条指令 replay()]
                ↓
GPU 时间线：
[GEMM][RMSNorm][GEMM][...]连续执行，无空闲
```

### CUDA Graph 的限制

使用 CUDA Graph 有一些重要限制，这解释了 FlashRT 很多设计决策：

1. **内存地址必须固定**：录制时使用的所有 GPU 内存地址，在重放时必须相同。这就是为什么 FlashRT 预先分配所有缓冲区，不在推理时做任何动态内存分配。

2. **不能有 Python 控制流**：if/for/while 等 Python 代码不能在 CUDA Graph 内部。这就是为什么流水线层只接受指针，不接受 tensor 对象——因为 tensor 对象可能触发 Python 控制流。

3. **prompt 长度变化需要重建**：如果 prompt 长度变了，sequence length 变了，就需要重新录制（因为缓冲区大小不同）。这就是为什么 `set_prompt()` 在检测到长度变化时会重建流水线。

### 性能提升量化

以 Pi0.5（RTX 5090）为例：

| 方式 | 延迟 | 说明 |
|------|------|------|
| 普通 PyTorch FP16 | ~150ms | 每层单独调度，BF16 GEMM |
| 普通 PyTorch FP8 | ~80ms | FP8 GEMM，但 Python 调度开销仍在 |
| FlashRT (CUDA Graph + FP8) | **17.58ms** | 零调度开销，融合内核 |

## 概念3：FlashAttention-2

### 普通注意力的问题

标准注意力计算（Scaled Dot-Product Attention）：

```python
# 输入：Q (seq, d_head), K (seq, d_head), V (seq, d_head)
# 标准实现
scores = Q @ K.T / sqrt(d_head)     # (seq, seq) — 这个矩阵很大！
weights = softmax(scores, dim=-1)    # (seq, seq)
output = weights @ V                 # (seq, d_head)
```

当 seq=512（本例中视觉编码器），注意力矩阵大小为 512×512 = 262144 个元素。这个矩阵需要在全局内存（HBM）中读写，非常慢。

### FlashAttention 的解决方案

FlashAttention 把 Q/K/V 分成小块（tile），在 GPU 的共享内存（SRAM）中完成计算，避免把大矩阵写到 HBM：

```
标准注意力（HBM 读写量）：
  写入 scores:   O(seq²)
  读取 scores:   O(seq²)
  写入 weights:  O(seq²)
  读取 weights:  O(seq²)
  合计：O(seq²) HBM 访问

FlashAttention（HBM 读写量）：
  只需读 Q/K/V 和写 O：O(seq × d_model)
  中间结果在 SRAM 中计算，不写回 HBM
```

对于 seq=512, d_model=1152，这大约是 ~44×的内存访问减少。

### FlashAttention-2 的改进

FlashAttention-2 在 FA1 基础上改进了工作划分策略，使 GPU 利用率从 ~25% 提升到 ~73%。主要改进：
1. 减少 softmax 的 rescale 操作
2. 并行化序列维度（而不只是批次/头维度）
3. 更好的共享内存使用

FlashRT 在 `csrc/attention/` 中供应商内置了 FA2（版本 2.7.4.post1），确保与 FlashRT 的 FP8 和 CUDA Graph 工作流完全兼容。

## 概念4：GQA（分组查询注意力）

### 标准多头注意力（MHA）

```
Query 头数 = Key 头数 = Value 头数 = n_heads

例：Gemma-2B Encoder
  n_heads = 8
  n_kv_heads = 8
  Q: (seq, 8, 256)  — 每个 Q 头有 256 维
  K: (seq, 8, 256)  — 每个 K 头有 256 维
  V: (seq, 8, 256)  — 每个 V 头有 256 维
```

### 分组查询注意力（GQA）

```
Query 头数 > Key 头数 = Value 头数

例：Gemma-2B（实际是 GQA）
  n_heads = 8     ← Q 头数
  n_kv_heads = 4  ← K/V 头数（是 Q 头数的一半）

  Q: (seq, 8, 256)
  K: (seq, 4, 256)   ← K 的每个头要处理 2 个 Q 头的查询
  V: (seq, 4, 256)

KV 缓存大小减半！
```

**对推理的意义**：
- KV 缓存更小 → 内存占用更少
- 不牺牲模型质量（实验证明 GQA 与 MHA 性能相当）
- 解码时读写 KV 缓存的带宽需求减半

## 概念5：RoPE（旋转位置编码）

### 为什么需要位置编码？

Transformer 的注意力机制本身没有位置信息——它只知道"哪个 token 与哪个 token 相似"，但不知道"它们的位置关系"。位置编码解决这个问题。

### RoPE 的工作原理

RoPE 把位置信息编码进 Query 和 Key 的旋转中：

```
普通注意力：score(q, k) = q · k

RoPE 注意力：score(q_i, k_j) = (R_θ·q_i) · (R_θ·k_j)
             其中 R_θ 是根据位置 i/j 计算的旋转矩阵
```

位置差 i-j 对 score 的贡献是确定的，这使得模型能外推到训练时没见过的序列长度。

### FlashRT 中的 RoPE

视觉编码器的 RoPE 表在加载时预计算（因为 patch 位置固定）：
```python
# 加载时
rope_freqs = precompute_rope_freqs(max_seq_len=256, head_dim=128)
```

解码器的 RoPE 在每次设置 prompt 时更新（因为 prompt 长度可变）：
```python
# set_prompt() 时
decoder_rope = precompute_rope_freqs(prompt_len + chunk_size, head_dim=256)
```

## 概念6：扩散模型（Diffusion Model）

### 为什么用扩散模型生成动作？

机器人操作是"多模态"的——给定同样的图像和指令，可能有多种合理的动作序列（不同的抓取方式都是正确的）。普通的回归模型（输出确定值）不善于建模这种一对多的关系。

扩散模型天然支持多模态分布：每次推理从不同的随机噪声出发，可能得到不同但都合理的动作序列。

### 扩散推理的工作原理

```
初始：纯随机噪声 x_T ~ N(0, I)  (chunk_size=10, action_dim=32)

步骤1（t=T）：
  网络预测"去噪方向" ε_T
  x_{T-1} = x_T - step_size × ε_T

步骤2（t=T-1）：
  网络预测"去噪方向" ε_{T-1}
  x_{T-2} = x_{T-1} - step_size × ε_{T-1}

...（重复 10 次）...

步骤10（t=1）：
  x_0 = 真实动作

每一步去噪时，网络都会"看"编码器的 K/V 缓存，
确保生成的动作符合图像和文字指令。
```

### AdaRMSNorm 与时间步条件

扩散网络需要知道当前处于哪个去噪步，才能做出正确的预测。FlashRT 用 AdaRMSNorm 来传递这个信息：

```
时间步 t → MLP → 调制参数 (scale, shift, gate)
                      ↓
x_norm = RMSNorm(x) * (1 + scale) + shift
output = gated_residual(x, x_norm, gate)
```

每个解码器层都有自己的 AdaRMSNorm，每层的 scale/shift/gate 都不同——这样模型可以为不同步骤、不同层提供不同的"条件"。

**关键优化**：这些调制参数只依赖于 prompt，不依赖于输入动作。因此 FlashRT 在 `set_prompt()` 时就把所有 10×18 层的调制参数全部预计算好，存成：

```
precomputed_styles: (num_steps=10, num_layers=18, chunk_size, 3*decoder_dim)
```

推理时直接取出对应位置的参数，零计算开销。

## 概念7：NVFP4（Blackwell FP4 量化）

RTX 5090（SM120，Blackwell 架构）支持 FP4 计算，比 FP8 快一倍。

### FP4 的挑战

FP4 只有 4 位（1位符号 + 2位指数 + 1位尾数），动态范围极小（最大值 6.0），直接量化精度损失很大。

### 块缩放（Block-scaled Quantization）

NVFP4 使用"块缩放"解决精度问题：把权重分成 16 个一组的小块，每块有自己的缩放因子：

```
权重矩阵（举例）：[1.0, 2.0, 3.0, 4.0, ..., 50.0, 60.0, 70.0, 80.0, ...]
                  |--- 第1块(最大值≈4, s=4/6=0.67) ---|--- 第2块(最大值≈80, s=80/6=13.3) ---|

第1块量化：
  1.0 → round(1.0/0.67) = round(1.5) ≈ 2 → 反量化: 2×0.67 = 1.33  （误差 0.33）
  4.0 → round(4.0/0.67) = round(6.0) ≈ 6 → 反量化: 6×0.67 = 4.02  （误差 0.02）

第2块量化：
  50.0 → round(50.0/13.3) ≈ 4 → 反量化: 4×13.3 = 53.2  （误差 6%）
  80.0 → round(80.0/13.3) ≈ 6 → 反量化: 6×13.3 = 79.8  （误差 0.2%）
```

块缩放使得每个小块内的精度都被最大化，整体精度远高于全局单一缩放。

### AWQ（激活感知权重量化）

NVFP4 还结合 AWQ 技术：分析激活值的分布，对"重要的"权重通道使用更多位数的精度。这使得 FP4 量化后的模型质量接近 FP8。

## 概念小结

| 技术 | 解决的问题 | 代价 |
|------|-----------|------|
| FP8 量化 | 减少内存，加速 GEMM | 需要校准步骤 |
| CUDA Graph | 消除 CPU 调度开销 | 需要固定内存布局，不能有动态控制流 |
| FlashAttention-2 | 减少注意力计算的 HBM 访问 | 需要特殊内核，不通用 |
| GQA | 减少 KV 缓存大小 | 轻微精度权衡（通常可忽略） |
| RoPE | 位置编码，支持长序列外推 | 无明显代价 |
| 扩散模型 | 支持多模态动作分布 | 需要多步推理（但已通过优化压缩到10步） |
| NVFP4 | 进一步加速（Blackwell 专用） | 精度需要更仔细的量化策略 |
