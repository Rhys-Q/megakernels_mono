# Mirage MPK 深度问答（QA1）

## Q1：任务图的节点是什么？MPK 调度的是 kernel 还是 thread block？

**结论：MPK 调度的单位是一个 thread block（CTA）。**

---

### 详细解释

传统 CUDA 编程中，你这样启动一个 kernel：

```cuda
linear_kernel<<<grid_dim, block_dim>>>(A, B, C);
// GPU 会同时启动 grid_dim.x * grid_dim.y * grid_dim.z 个 thread block
// 每个 block 的 blockIdx 不同，各自计算输出矩阵的一个 tile
```

MPK 完全不同。整个 MegaKernel 只启动一次，而且 **grid_dim = 固定的 SM 数量**：

```cuda
// 运行时只有这两个 kernel launch（各一次）
worker_kernel<<<num_workers, WORKER_NUM_THREADS, smem>>>(config);
// 例如：worker_kernel<<<96, 256, 207KB>>>(config)
// 96 个 block，每个 block = 一个 Worker SM
```

每个 Worker SM 对应 MegaKernel 的一个 block（`blockIdx.x` = Worker ID）。这 96 个 block 在整个 LLM 推理期间**永远不退出**，持续循环取任务、执行、取下一个任务。

---

### "任务"和"thread block"的关系

当用户定义一个 Linear 层：

```python
mpk.linear_layer(..., grid_dim=(96, 1, 1), block_dim=(128, 1, 1))
```

MPK 编译器创建 **96 个 TaskDesc**，每个 TaskDesc 对应这个层的输出矩阵的 1/96。

运行时发生的事情：

```
Scheduler → 将 task_id[0..95] 推入各 Worker 的任务队列

Worker SM 0 → 取出 task_id[0] → 用自己的 256 个线程执行 "Linear tile 0"
Worker SM 1 → 取出 task_id[1] → 用自己的 256 个线程执行 "Linear tile 1"
...
Worker SM 95 → 取出 task_id[95] → 用自己的 256 个线程执行 "Linear tile 95"
```

每个 Worker SM 的 256 个线程（2 个 warp group）共同完成 **一个 tile** 的矩阵乘法。这等价于传统 Linear kernel 里的"一个 thread block"的工作量。

---

### 关键区别

| 对比点 | 传统 CUDA kernel | MPK 任务 |
|--------|-----------------|---------|
| 标识符 | `blockIdx.x` | `TaskDesc` 中的预计算指针偏移 |
| "分块"如何确定 | 运行时 `blockIdx * tile_size` | 编译时写入 TaskDesc 的 base_ptr |
| grid_dim 含义 | 实际 launch 的 block 数 | 创建多少个 TaskDesc |
| block_dim 含义 | 每个 block 的线程数 | Worker SM 用于此任务的线程数 |
| 并发度控制 | 由 GPU 自动调度 | 由 Scheduler 决定何时推入 Worker 队列 |

---

### 一个比喻

把 Worker SM 想象成工厂里的工人，`all_tasks[]` 是一张工单列表：
- 每张工单（TaskDesc）上写着：做什么（task_type）、用哪块原料（input_ptrs）、放到哪里（output_ptrs）
- 工人不需要知道"我是第几号工人"，只需按工单做事
- 调度员（Scheduler）决定什么时候把工单分发给哪个工人

这就是为什么 MPK 的"任务"本质上是 **thread block 级别的工作单元**。

## Q2：MPK 的 Event 同步是怎么实现的？依赖 CUDA 的什么同步机制？

**结论：MPK 完全不依赖 CUDA 的高级同步 API（无 cudaEventSynchronize、无 cudaStreamWaitEvent、无 __syncthreads 跨 block）。它完全基于 PTX 指令级别的原子操作 + 自旋等待（spin-wait），通过 acquire/release 内存序保证正确性。**

---

### 四个核心原语

MPK 的 event 系统使用四个 PTX 级别的原子操作（定义在 `mpk_atoms.cuh` 中）：

| 原语 | 作用 | PTX 等效 |
|------|------|---------|
| `ld_acquire_gpu_u64(ptr)` | 带 acquire 语义的 64 位加载（GPU 范围） | `ld.acquire.gpu.u64` |
| `ld_acquire_sys_u64(ptr)` | 带 acquire 语义的 64 位加载（系统范围，含跨 GPU）| `ld.acquire.sys.u64` |
| `atom_add_release_gpu_u64(ptr, val)` | 带 release 语义的原子加（GPU 范围）| `atom.add.release.gpu.u64` |
| `atom_cas_release_gpu_u64(ptr, old, new)` | 带 release 语义的 CAS（GPU 范围）| `atom.cas.release.gpu.u64` |

加上 `__nanosleep(10)`：让当前线程休眠约 10ns，减少内存总线压力（避免自旋过于激烈）。

---

### Event 触发流程（Worker 完成任务后）

```cpp
// Worker 完成任务后，触发 trigger_event：
if (threadIdx.x == 0) {
    EventId event_id = task_desc->trigger_event;
    size_t event_index = get_event_position_index(event_id);

    // 原子加：将事件计数器 +1，返回旧值
    EventCounter count = atom_add_release_gpu_u64(
        &config.all_event_counters[event_index], 1);
    // ↑ release 语义：保证此 Worker 所有计算结果在这步之前对其他线程可见

    int num_triggers = config.all_event_num_triggers[event_index];
    if ((count + 1) == num_triggers * iteration_num) {
        // 所有依赖的 Worker 都完成了 → 事件就绪
        // 将事件 ID 写入 Scheduler 的事件队列
        size_t pos = atom_add_release_gpu_u64(&sched_queue_next_free_event_id, 1);
        st_relaxed_gpu_u64(&sched_queues[sched_id][pos % queue_len], event_index);
        // CAS 循环，确保队列有序（防止乱序）
        do {
            old = atom_cas_release_gpu_u64(&sched_queue_last_ready_event_id, pos, pos + 1);
        } while (old != pos);
    }
}
```

---

### Event 等待流程（Worker 等待依赖）

```cpp
// Worker 取到任务后，等待其 dependent_event 就绪：
if (threadIdx.x == 0) {
    if (task_desc->dependent_event != EVENT_INVALID_ID) {
        EventCounter needed_counts = num_triggers * iteration_num;
        EventCounter actual_counts = 0;
        while (actual_counts < needed_counts) {
            // acquire 语义加载：保证能看到所有 release 之前的写入
            actual_counts = ld_acquire_sys_u64(&config.all_event_counters[event_index]);
            __nanosleep(10);  // 让出内存总线，每次等待约 10ns
        }
    }
}
__syncthreads();  // ← 注意：这里的 __syncthreads 是 block 内部同步，
                  //   确保 block 内的 256 个线程都等待完再开始执行任务
```

---

### Scheduler 等待事件队列

```cpp
// Scheduler 自旋等待事件队列里有新事件：
while (cur_event_pos == last_event_pos) {
    last_event_pos = ld_acquire_gpu_u64(
        &config.sched_queue_last_ready_event_id[sched_id]);
    __nanosleep(10);
}
```

---

### 内存序的关键作用

**acquire/release** 保证了正确的内存可见性：

```
Worker 完成 Linear 计算（写 output_ptr 内存）
    ↓
atom_add_release_gpu_u64(event_counter, 1)
  "release" 保证：上面的 output_ptr 写入，在 acquire 方看来已经完成
    ↓
（传播到 event_counter）
    ↓
ld_acquire_sys_u64(event_counter) 在 Attention Worker 中返回新值
  "acquire" 保证：此读之后，能看到 release 之前的所有写入
    ↓
Attention Worker 安全读取 Linear 的输出内存
```

如果不用 acquire/release，两个 Worker 之间没有内存顺序保证，Attention Worker 可能读到 Linear 尚未写完的旧数据。

---

### 为什么不用 CUDA stream 或 cudaEvent？

原因：**它们是 host（CPU）API，无法在 GPU kernel 内部调用**。

CUDA 中：
- `cudaStreamWaitEvent(stream, event)` — CPU 调用，让一个 stream 等待另一个的事件
- `cudaEventRecord(event, stream)` — CPU 调用，在 stream 中插入一个时间戳事件

这些必须由 CPU 驱动来触发，不能在 GPU kernel 内部使用。MPK 需要的是 **GPU 内部的同步**，因此完全用软件（原子 + 自旋）实现。

唯一例外：**多 GPU 的 NVSHMEM 事件** 使用 `nvshmem_signal_wait_until`，这是 NVSHMEM 提供的跨 GPU 信号量等待。

## Q3：MPK 的 kernel 是怎么生成的？如何将一个 kernel 拆成多个 thread block 单独调度？

**结论：编译器在生成 CUDA 代码时，为每个 TaskDesc 静态计算出其对应的内存偏移量（offset），并将这个偏移量写入 `init_persistent_kernel()` 的初始化代码中。GPU 启动时执行这个初始化代码，把每个任务的 `base_ptr` 设置为 `tensor_start + bid_offset`。此后 Worker 直接读 `task_desc.input_ptrs`，无需任何运行时偏移计算。**

---

### 传统 CUDA kernel 如何"分块"

传统方式，每个 block 在运行时计算自己要处理哪一块：

```cuda
__global__ void linear_kernel(float *A, float *B, float *C, int N) {
    int tile_col = blockIdx.x * TILE_SIZE;   // 运行时计算
    // ...
    C[row][tile_col + local_col] = result;
}
// launch: linear_kernel<<<96, 128>>>(A, B, C, 4096)
// 每个 block 的 blockIdx.x 不同，所以各自算不同的 tile
```

---

### MPK 如何解决同样的问题

MPK 的 Worker SM 没有意义上的 `blockIdx.x`（它的 `blockIdx.x` 是 Worker ID，不是"第几个 tile"）。

解决方案：**在生成的 CUDA 代码的 `init_persistent_kernel()` 函数中，对每个任务 (bid.x, bid.y, bid.z) 静态计算出其内存偏移，并直接写入 `all_tasks[task_id].inputs[i].base_ptr`。**

---

### 编译器生成的初始化代码（简化示意）

编译器（`src/kernel/runtime.cc`）生成如下的初始化代码片段（这是生成的 CUDA 代码，不是编译器本身的代码）：

```cuda
// 以下是生成在 kernel.cu 中的 init_persistent_kernel() 片段
// 针对第 0 层的 QKV Linear 层（grid_dim=(96,1,1)）

// bid.x = 0 对应的任务
{
    TensorDesc input0;
    input0.base_ptr = (char*)rmsnorm_out + 0;       // offset = 0 * tile_size
    input0.dim = {8, 4096}; input0.stride = {4096, 1};
    TensorDesc input1;
    input1.base_ptr = (char*)w_qkv + 0;             // output 的前 64 列
    TensorDesc output0;
    output0.base_ptr = (char*)attn_in + 0;           // offset = 0 * 64 * 2 bytes
    all_tasks[task_id_0].inputs[0] = input0;
    all_tasks[task_id_0].outputs[0] = output0;
}

// bid.x = 1 对应的任务
{
    TensorDesc output0;
    output0.base_ptr = (char*)attn_in + 64 * 2;     // offset = 1 * 64列 * sizeof(bf16)
    all_tasks[task_id_1].outputs[0] = output0;
    // ...
}

// bid.x = 95 对应的任务
{
    TensorDesc output0;
    output0.base_ptr = (char*)attn_in + 95 * 64 * 2; // 最后一个 tile
    all_tasks[task_id_95].outputs[0] = output0;
}
```

实际的编译器代码（`runtime.cc`）生成这些偏移的逻辑是：

```cpp
// 编译器中的代码（生成初始化逻辑）
if (input_map.x >= 0) {
    size_t block_size = io_desc.tensor.dim[input_map.x] / bgraph.grid_dim.x;
    // block_size = 5120 / 96 ≈ 53 (每个 tile 的列数)
    offset += block_size * bid.x * io_desc.tensor.stride[input_map.x];
    // offset = 53 * bid.x * 1 (列方向步长为1)
}
tgbody.e("input$.base_ptr = static_cast<char*>($) + $;",
         i, io_desc.name, offset * sizeof(bfloat16));
// 生成: input0.base_ptr = (char*)attn_in + 53 * bid.x * 2;
```

### 完整的三阶段分工

```
阶段 1：Python 编译期（CPU，nvcc 编译时）
  ├── 分析 grid_dim=(96,1,1)，计算 block_size = output_dim / 96
  ├── 生成循环代码：
  │     for bid.x in range(96):
  │         offset = block_size * bid.x * stride
  │         all_tasks[task_id].outputs[0].base_ptr = tensor_ptr + offset
  └── 输出：kernel.cu（含上述初始化代码）+ nvcc 编译 → kernel.so

阶段 2：GPU 初始化期（运行时，init_persistent_kernel 被调用）
  ├── 执行上述初始化代码（此时 tensor_ptr 是真实的 GPU 内存地址）
  ├── 完成后：all_tasks[0..95] 的 base_ptr 分别指向 attn_in 的不同列段
  └── 不需要任何 CPU 参与（这段代码在 GPU 上运行）

阶段 3：GPU 计算期（Worker 执行任务）
  ├── Worker SM 取出 TaskDesc，直接读 input_ptrs[i] / output_ptrs[i]
  ├── 不做任何偏移计算，直接用指针访问正确的内存位置
  └── 等效于传统 kernel 里 "blockIdx.x = task_id" 的效果
```

---

### 为什么 base_ptr 不在 Python 编译时确定？

因为 `tensor_ptr`（GPU 内存地址）在 Python 编译时还不知道！

```python
# Python 编译时：
y = mpk.new_tensor(dims=(8, 4096), ..., name="attn_in")
# 此时只有"名字"和"形状"，没有实际 GPU 地址

mpk.compile(output_dir="./kernels")
# 编译期只生成"init代码：把名字 'attn_in' 对应的地址 + offset 填入 TaskDesc"

# Python 运行时（调用 mpk() 之前）：
mpk()
# → init_persistent_kernel() 被调用，此时 'attn_in' 的 GPU 地址已知
# → 将 tensor地址 + 各bid的offset 填入 all_tasks 数组
# → 然后启动 worker_kernel 和 scheduler_kernel
```

---

### 总结对比

| 问题 | 传统 kernel | MPK |
|------|------------|-----|
| "哪个 block 处理哪个 tile" | 运行时 `blockIdx.x * tile_size` | 初始化时写入 `all_tasks[i].base_ptr` |
| 确定时机 | 每次 kernel 执行时计算 | kernel.so 加载后 init 一次 |
| Worker 需要知道自己的 ID 吗 | 需要（blockIdx） | 不需要（直接读 TaskDesc 指针） |
| 开销 | 每次 kernel 都要计算偏移 | 只初始化一次，之后零开销 |
