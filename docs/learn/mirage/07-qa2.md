# Mirage MPK 深度问答（QA2）

## Q1：Mirage 支持动态 Shape 吗？

**结论：不支持真正的动态 Shape。MPK 使用"静态上限 + 运行时动态填充"的模式。**

---

### 什么是"静态上限"

MPK 在编译时（nvcc 编译阶段）通过宏定义固定最大形状：

```cpp
// 编译命令中的 -D 参数（会变成 C++ 宏定义）
-DMPK_MAX_NUM_BATCHED_REQUESTS=8   // 最多同时处理 8 个请求
-DMPK_MAX_NUM_BATCHED_TOKENS=8     // 最多 8 个 token/次迭代
-DMPK_MAX_SEQ_LENGTH=512           // 序列最长 512 个 token
-DMPK_MAX_NUM_PAGES=16             // KV Cache 最多 16 页
-DMPK_PAGE_SIZE=4096               // 每页 4096 个 token
```

这些数字直接参与 CUDA 代码的编译优化（循环展开、寄存器分配等），一旦编译就**无法更改**。

### 什么可以在运行时变化

| 参数 | 是否可以运行时变化 | 说明 |
|------|-----------------|------|
| 实际 batch 中的请求数 | ✅ 可以 | 0 ~ MAX_NUM_BATCHED_REQUESTS |
| 每个请求的当前步骤（step）| ✅ 可以 | 读取 `step[]` 数组 |
| 每个请求的页表（KV Cache）| ✅ 可以 | 由页面分配器动态管理 |
| 张量的 shape（形状） | ❌ 不可以 | 编译时固定 |
| 模型权重 | ❌ 不可以 | 指针在 init 时确定，不能热替换 |

### 实际例子

如果你想支持不同的 batch size：

```python
# 方案 A：使用最大 batch size 编译，实际用更小的 batch（有些 SM 空闲）
mpk = mi.PersistentKernel(max_num_batched_tokens=64)  # 编译时上限 64
# 实际运行时可以只填 1 个请求，其余请求槽空着

# 方案 B：不同 batch size 各编译一个版本（需要多次 compile()）
mpk_bs1  = mi.PersistentKernel(max_num_batched_tokens=1)
mpk_bs64 = mi.PersistentKernel(max_num_batched_tokens=64)
```

### 为什么不支持动态 Shape？

这是有意识的**性能取舍**：

- 编译时已知 shape → 循环次数是常量 → 编译器可以完全展开循环
- 编译时已知 shape → Shared memory 布局固定 → 无需运行时计算偏移
- 编译时已知 shape → 寄存器分配静态优化

代价是：如果预分配的 max shape 远大于实际使用，有些 SM 会有空闲浪费。

## Q2：MPK 的编译产物是什么？编译产物 ABI 是什么？

### 编译产物文件

执行 `mpk.compile(output_dir="./kernels")` 后产生：

```
kernels/
├── test_rank0.cu                           # CUDA 源代码（可用于调试）
├── task_graph_rank0.json                   # 任务图描述（JSON）
├── mpk_launcher_rank0.cpython-311-x86_64-linux-gnu.so  # 编译好的共享库
└── kernel_metadata_rank0.json             # 兼容性元数据
```

### 核心产物：.so 文件的 ABI

`.so` 文件是一个 **Python C 扩展模块**，模块名为 `__mirage_launcher`，对外暴露 4 个函数：

```c
// Python 模块接口（由 HARD_CODE 段定义在 persistent_kernel.py 中）
static PyMethodDef ModuleMethods[] = {
    {"init_func",         init_func,         METH_VARARGS, "initialize persistent kernel"},
    {"init_request_func", init_request_func, METH_VARARGS, "initialize request resources"},
    {"launch_func",       launch_func,       METH_VARARGS, "launch persistent kernel"},
    {"finalize_func",     finalize_func,     METH_VARARGS, "finalize persistent kernel"},
};
```

每个函数的 C 签名（内部调用）：

| Python 函数 | 调用的 C 函数 | 作用 |
|------------|-------------|------|
| `init_func(meta_list, profiler_buf, rank, num_workers, ..., tensor_names, tensor_ptrs, json_path)` | `init_persistent_kernel(...)` | 分配 GPU 资源，初始化任务图 |
| `init_request_func()` | `init_request_resources()` | 初始化请求队列、页面分配器 |
| `launch_func(stream_ptr)` | `launch_persistent_kernel(stream)` | 启动 MegaKernel（阻塞直到完成） |
| `finalize_func()` | `finalize_persistent_kernel()` | 释放所有 GPU 资源 |

### task_graph.json 的结构

```json
{
  "tasks": [
    {
      "task_type": 10,   // TASK_BEGIN_TASK_GRAPH
      "variant_id": 0,
      "trigger_event": 12345678,
      "dependent_event": 18446744073709551614,  // EVENT_INVALID_ID
      "inputs": [],
      "outputs": []
    },
    {
      "task_type": 160,  // TASK_EMBEDDING_HOPPER
      "variant_id": 0,
      "trigger_event": ...,
      "inputs": [
        {"base_ptr": "input_token", "offset": 0, "data_type": 8, "dims": [8,1], "strides": [1,1]},
        {"base_ptr": "embed_tokens", "offset": 0, "data_type": 2, "dims": [8,4096], "strides": [4096,1]}
      ],
      "outputs": [
        {"base_ptr": "embed_out", "offset": 0, ...}
      ]
    },
    // ... 数千个任务
  ],
  "events": [
    {"event_type": 901, "num_triggers": 1, "first_task": 2, "last_task": 3},
    // ...
  ]
}
```

注意：`base_ptr` 字段存的是**张量名字**（字符串），而非实际地址。实际的 GPU 指针在 `init_persistent_kernel()` 被调用时、通过 `model_tensors` 字典查找后填入。

### kernel_metadata.json 的结构（兼容性校验）

```json
{
  "world_size": 1,
  "mpi_rank": 0,
  "num_workers": 96,
  "num_local_schedulers": 4,
  "max_num_batched_requests": 8,
  "max_num_batched_tokens": 8,
  "max_seq_length": 512,
  "max_num_pages": 16,
  "page_size": 4096,
  "mode": "offline",
  "target_cc": 90
}
```

`load_mpk_kernel()` 加载已有 .so 时会比对这个 JSON，确保 kernel 和当前配置兼容。

## Q3：MPK Runtime 详细工作原理

### 3.1 加载编译产物

```python
import importlib.util
spec = importlib.util.spec_from_file_location("__mirage_launcher", so_path)
mod = importlib.util.module_from_spec(spec)
spec.loader.exec_module(mod)           # ← 执行 dlopen + PyInit___mirage_launcher()
self.init_func   = getattr(mod, "init_func")
self.launch_func = getattr(mod, "launch_func")
```

这就是标准的 Python `ctypes.CDLL` / 扩展模块加载：`spec.loader.exec_module(mod)` 等价于 `dlopen(so_path)` 并调用 `PyInit___mirage_launcher()`，注册上述 4 个函数。

---

### 3.2 init_func 初始化序列

```python
self.init_func(
    meta_tensors_ptr,    # [step_ptr, tokens_ptr, ..., snapshot_ptr] — 11 个元数据指针
    profiler_buffer_ptr,
    rank, num_workers, num_local_schedulers, num_remote_schedulers,
    max_seq_length, total_num_requests, eos_token_id, ...,
    model_tensor_names,  # ["embed_tokens", "layer_0_q_proj", ...]
    model_tensor_ptrs,   # 对应的 GPU 内存地址
    json_path,           # task_graph.json 的路径（用于 runtime 加载任务图）
)
```

`init_persistent_kernel()` 在 CPU 上顺序执行：

```
1. 构建 global_model_tensors：{名字 → GPU 地址} 的哈希表

2. 填充 global_runtime_config 的元数据指针字段（step, tokens, input_tokens...）

3. （如果 USE_NVSHMEM）初始化 NVSHMEM：nvshmemx_init_attr()

4. 调用 _init_persistent_kernel()（transpiler 生成的函数）：
   ├── 遍历所有任务 (bid.x, bid.y, bid.z)
   ├── 查找 global_model_tensors["embed_tokens"] 得到 GPU 地址
   ├── 计算每个 task 的 base_ptr = tensor_addr + offset
   ├── 填充 all_fulltasks[] 和 all_events[]
   └── 填充 first_tasks[]（每次迭代的起始任务）

5. 将 all_fulltasks[] 转换为 all_tasks[]（FullTaskDesc → TaskDesc）

6. GPU 内存分配（cudaMalloc）：
   ├── all_tasks[]        → GPU  (cudaMemcpy)
   ├── all_events[]       → GPU  (cudaMemcpy)
   ├── worker_queues[96]  → GPU  (每个 worker 一个环形队列，深度 8192)
   ├── sched_queues[4+1]  → GPU  (每个 scheduler 一个事件队列，深度 4096)
   ├── event_counters[]   → GPU  (所有事件的原子计数器)
   └── event_num_triggers[]→ GPU  (每个事件需要触发几次才就绪)

7. 创建 CUDA stream 和 event：
   ├── worker_stream    (NonBlocking)
   ├── scheduler_stream (NonBlocking)
   ├── prepare_done_event
   ├── worker_done_event
   └── scheduler_done_event

8. 调用 init_kernel<<<1,128>>>()：初始化请求队列、页面分配器
```

初始化结束后，GPU 上已经有：
- 静态的 `all_tasks[]` 和 `all_events[]` 数组
- 空的 `worker_queues[]` 和 `sched_queues[]`（还没有任务/事件）

---

### 3.3 launch_func 启动序列

```python
self.launch_func(stream_ptr)  # → launch_persistent_kernel(stream)
```

`launch_persistent_kernel()` 分 3 步：

**第 1 步：prepare_kernel 重置队列**

```cuda
// 重置所有队列的读写指针为 0
prepare_kernel<<<num_workers, 128, 0, default_stream>>>(config, end_event_pos);
cudaEventRecord(prepare_done_event, default_stream);
```

关键操作：将 `EVENT_END_OF_TASK_GRAPH` 的事件 ID 写入 `sched_queues[0][0]`，并把 `sched_queue_last_ready_event_id[0] = 1`。这是整个系统的"第一个触发信号"。

**第 2 步：同时启动 Worker 和 Scheduler**

```cuda
// Worker stream：启动 96 个 Worker block
cudaStreamWaitEvent(worker_stream, prepare_done_event, 0);
worker_kernel<<<96, 256, 207KB, worker_stream>>>(config);
cudaEventRecord(worker_done_event, worker_stream);

// Scheduler stream：启动 4 个 Scheduler block
cudaStreamWaitEvent(scheduler_stream, prepare_done_event, 0);
scheduler_kernel<<<4, 32, 0, scheduler_stream>>>(config);
cudaEventRecord(scheduler_done_event, scheduler_stream);

// 让 default_stream 等待两者都完成
cudaStreamWaitEvent(default_stream, worker_done_event, 0);
cudaStreamWaitEvent(default_stream, scheduler_done_event, 0);
```

两个 kernel 在**不同的 CUDA stream** 上并发运行，GPU 调度器确保它们真正同时执行。

**第 3 步：Python 侧等待**（对于离线模式）

`launch_func` 返回后，Python 的 `mpk()` 调用等待 `default_stream` 完成，即等待 Worker 和 Scheduler 都退出。

---

### 3.4 多 Scheduler 协同工作原理

以 `num_local_schedulers=4`、`num_workers=96` 为例。

**Scheduler 的分工**：

`scheduler_kernel<<<4, 32>>>`，4 个 SM block，每个 block 32 个线程（1 个 warp）。每个 warp 的 `threadIdx.x=0` 号线程运行一个 Scheduler 逻辑：

```
Scheduler 0 (blockIdx.x=0)：管理 Worker 0..23
Scheduler 1 (blockIdx.x=1)：管理 Worker 24..47
Scheduler 2 (blockIdx.x=2)：管理 Worker 48..71
Scheduler 3 (blockIdx.x=3)：管理 Worker 72..95
```

还有一个"全局队列" `sched_queues[4]`（`sched_id = num_schedulers = 4`），所有 Scheduler 都会监听它（用于大批量任务的分发）。

**Scheduler 的任务分配（轮询）**：

```cpp
// 每个 Scheduler 内部维护一个 next_worker 指针，轮询分配
int next_worker = my_first_worker;  // 初始指向自己管辖的第一个 worker

// 分配任务时：
worker_queues[next_worker] ← 推入 task_id
next_worker = (next_worker == my_last_worker - 1) ? my_first_worker : next_worker + 1;
// 即：轮流分配给 my_first_worker, my_first_worker+1, ..., my_last_worker-1, 然后循环
```

**事件路由**：

Worker 完成任务后，将事件推给哪个 Scheduler 队列？

```cpp
int sched_id = worker_id / (num_workers / num_local_schedulers);
// Worker 0..23 → Scheduler 0
// Worker 24..47 → Scheduler 1
// 等
```

如果事件类型是 `EVENT_LAUNCH_MASSIVE_TASKS`（大规模并行任务），则推给全局队列（所有 Scheduler 分摊处理）。

### 3.5 用简单 Demo 模型说明：3 层 MLP

假设模型只有 3 层：`Linear_0 → ReLU → Linear_1 → ReLU → Linear_2`。

设 `grid_dim=(4,1,1)` 表示每层用 4 个 Worker 并行。

**编译后的任务图（all_tasks[]）**：

```
task_id=1: TASK_BEGIN_TASK_GRAPH    （无输入/输出，依赖无，触发→event_1）
task_id=2: TASK_LINEAR（tile_0/4）  （依赖→event_1，触发→event_2）
task_id=3: TASK_LINEAR（tile_1/4）  （依赖→event_1，触发→event_2）
task_id=4: TASK_LINEAR（tile_2/4）  （依赖→event_1，触发→event_2）
task_id=5: TASK_LINEAR（tile_3/4）  （依赖→event_1，触发→event_2）
task_id=6: TASK_RELU（tile_0/4）    （依赖→event_2，触发→event_3）
task_id=7: TASK_RELU（tile_1/4）    （依赖→event_2，触发→event_3）
task_id=8: TASK_RELU（tile_2/4）    （依赖→event_2，触发→event_3）
task_id=9: TASK_RELU（tile_3/4）    （依赖→event_2，触发→event_3）
// ... Linear_1, ReLU, Linear_2 类似
task_id=25: TASK_ARGMAX             （依赖→event_11，触发→event_12）
task_id=26: EVENT_END_OF_TASK_GRAPH（特殊任务，触发→终止或下一次迭代）
```

**事件表（all_events[]）**：

```
event_1: type=LAUNCH_TASKS, num_triggers=1, tasks=[2..5]
         （BEGIN_TASK_GRAPH 完成1次 → 触发 Linear_0 的 4 个任务）

event_2: type=LAUNCH_TASKS, num_triggers=4, tasks=[6..9]
         （4 个 Linear_0 任务全完成 → 触发 ReLU 的 4 个任务）

event_3: type=LAUNCH_TASKS, num_triggers=4, tasks=[10..13]
         （4 个 ReLU 任务全完成 → 触发 Linear_1 的 4 个任务）
...
event_12: type=END_OF_TASK_GRAPH, num_triggers=1, tasks=[]
```

**运行时第一次迭代详细过程**：

```
[CPU] prepare_kernel 推入 EVENT_END_OF_TASK_GRAPH 到 sched_queues[0]
[CPU] 启动 worker_kernel 和 scheduler_kernel

[Scheduler 0] 收到 EVENT_END_OF_TASK_GRAPH (iteration=0)
  → prepare_next_batch(): 准备第一批输入 tokens
  → 推入 compute_task_id(iter=1, pos=1)（BEGIN_TASK_GRAPH）到 Worker_0 的队列
  
[Worker 0] 取出 task_id=(iter=1, pos=1)
  → task_type = TASK_BEGIN_TASK_GRAPH
  → 等待 dependent_event（EVENT_INVALID → 无需等待）
  → 执行：什么都不做（no-op）
  → 原子加 event_counters[event_1] += 1
  → event_1 计数器 = 1 == num_triggers(1) → 触发!
  → 将 event_1 推入 sched_queues[0]

[Scheduler 0] 收到 event_1 (type=LAUNCH_TASKS)
  → 取出 tasks=[2,3,4,5]（Linear_0 的 4 个 tile）
  → 轮询分配到 Worker_0..3 的队列：
      Worker_0 ← task_id=(iter=1, pos=2)
      Worker_1 ← task_id=(iter=1, pos=3)
      Worker_2 ← task_id=(iter=1, pos=4)
      Worker_3 ← task_id=(iter=1, pos=5)

[Worker 0] 取出 task_id=(iter=1, pos=2) → Linear_0_tile_0
  → 等待 dependent_event=event_1
  → event_counters[event_1] = 1 = 1*1 ✓（已就绪）
  → 执行矩阵乘法（计算输出的第 0/4 列段）
  → 触发 event_2：event_counters[event_2] += 1

[Worker 1] 取出 task_id=(iter=1, pos=3) → Linear_0_tile_1
  → 并行执行（与 Worker_0 同时）
  → 触发 event_2：event_counters[event_2] += 1

[Worker 2], [Worker 3] 类似...

[当所有 4 个 Linear_0 任务完成]
  event_counters[event_2] = 4 = 4*1 → 触发!
  最后完成的 Worker 将 event_2 推入 sched_queues[?]

[Scheduler ?] 收到 event_2 → 分配 4 个 ReLU 任务 → ...

（重复上述过程，直到 ARGMAX 完成）

[Argmax 完成]
  → 触发 event_12 (END_OF_TASK_GRAPH)
  → Scheduler 0 收到 event_12
  → prepare_next_batch()：检查是否达到 EOS 或 max_seq_length
  → 如果继续：推入 compute_task_id(iter=2, pos=1)，开始下一次 decode
  → 如果停止：terminate_schedulers()，向所有 Worker 队列推入 TERMINATE 任务
  
[所有 Worker] 收到 TERMINATE → 退出循环
[所有 Scheduler] 收到 termination_event → 退出循环
[CPU 侧] cudaStreamWaitEvent 解除阻塞 → launch_func() 返回 → mpk() 返回
```

### 3.6 任务图如何传给 Scheduler？

**没有"传给"这回事——Scheduler 和任务图在同一块 GPU 内存里**。

`init_persistent_kernel()` 把所有任务和事件都 `cudaMemcpy` 到 GPU global memory：

```
GPU global memory:
┌─────────────────────────────────────────────────────────┐
│  all_tasks[]      — TaskDesc 数组，供 Worker 查找        │
│  all_events[]     — EventDesc 数组，供 Scheduler 查找    │
│  event_counters[] — 每个事件的触发计数器（原子变量）      │
│  worker_queues[96]— 每个 Worker 的任务 ID 队列           │
│  sched_queues[5]  — 每个 Scheduler 的事件 ID 队列        │
└─────────────────────────────────────────────────────────┘
```

- Scheduler 通过读自己的 `sched_queues[sched_id]` 知道哪些事件就绪
- 通过查 `all_events[event_id]` 知道事件关联的任务范围
- 通过写 `worker_queues[worker_id]` 把任务 ID 分发给 Worker
- Worker 通过读 `worker_queues[worker_id]` 取任务
- 通过查 `all_tasks[task_id]` 得到完整的 TaskDesc（包含函数类型和输入输出指针）

整个过程完全在 GPU 内存里进行，CPU 只需等待 `cudaStreamWaitEvent` 返回。

### 3.7 总结：Runtime 全景图

```
编译产物加载：dlopen(.so) → 4个函数指针
                                │
                                ▼
init_func() ─────────────── CPU 侧初始化
  _init_persistent_kernel()    ← 填充 TaskDesc (base_ptr = tensor_addr + offset)
  cudaMemcpy(all_tasks → GPU)  ← 任务图上传到 GPU
  cudaMemcpy(all_events → GPU) ← 事件图上传到 GPU
  cudaMalloc(worker_queues)    ← 每个 Worker 独立队列
  cudaMalloc(sched_queues)     ← 每个 Scheduler 独立队列
  init_kernel<<<1,128>>>()     ← GPU 初始化请求队列和页面分配器
                                │
                                ▼
launch_func() ──────────── 异步启动两个持久化 Kernel
  prepare_kernel()             ← 重置队列，推入第一个 EVENT_END_OF_TASK_GRAPH
  worker_kernel<<<96,256>>>()  ← Worker stream（异步）
  scheduler_kernel<<<4,32>>>() ← Scheduler stream（异步）
  cudaStreamWaitEvent()        ← 等待两者都完成
                                │
                                ▼
GPU 内部自循环（无 CPU 参与）：
  Scheduler: 监听 sched_queues → 解析 EventDesc → 轮询分配任务到 worker_queues
  Worker:    监听 worker_queues → 读 TaskDesc → spin-wait dependent_event
             → 执行计算 → atomicAdd trigger_event
             → 若触发条件满足 → 通知 Scheduler
                                │
                                ▼
所有请求完成 → terminate → worker/scheduler kernel 退出
  → cudaStreamWaitEvent 解除阻塞
  → launch_func 返回
  → Python mpk() 返回
```
